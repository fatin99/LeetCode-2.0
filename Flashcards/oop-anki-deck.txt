Name the four pillars of OOP.	Encapsulation, Abstraction, Inheritance, Polymorphism.	concepts
What is encapsulation?	Bundling data and the methods that operate on it within a class, and restricting direct access to internal state (typically via access modifiers).	concepts
What is abstraction?	Exposing only essential interfaces and hiding implementation details — callers depend on what an object does, not how.	concepts
What is inheritance?	A mechanism by which a subclass derives properties and behavior from a superclass, enabling code reuse.	concepts
What is polymorphism?	The ability for different classes to be treated through a common interface — the same call can dispatch to different implementations based on the runtime type.	concepts
Compile-time vs. runtime polymorphism.	Compile-time: method overloading (resolved by signature). Runtime: method overriding (resolved by actual object type via dynamic dispatch).	concepts
What is method overloading?	Multiple methods in the same class with the same name but different parameter lists.	concepts
What is method overriding?	A subclass providing its own implementation of a method already defined in its superclass — invoked via dynamic dispatch.	concepts
Composition vs. inheritance — what's the principle?	"Favor composition over inheritance." Composition (has-a) is more flexible than inheritance (is-a) and avoids tight coupling and fragile hierarchies.	concepts
What is an abstract class?	A class that cannot be instantiated and is meant to be subclassed; may contain abstract methods (without implementation) that subclasses must implement.	concepts
What is an interface?	A contract specifying methods a class must implement. Languages like Java use it for multiple inheritance of types.	concepts
What does the SOLID acronym stand for?	Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion.	solid
SOLID: Single Responsibility Principle.	A class should have only one reason to change — it should do one thing.	solid
SOLID: Open/Closed Principle.	Software entities should be open for extension but closed for modification — add new behavior by subclassing or composing, not by editing existing code.	solid
SOLID: Liskov Substitution Principle.	Subtypes must be substitutable for their base types without altering correctness — a subclass shouldn't break its parent's contracts.	solid
SOLID: Interface Segregation Principle.	Prefer many small client-specific interfaces over one large general-purpose one — clients shouldn't depend on methods they don't use.	solid
SOLID: Dependency Inversion Principle.	Depend on abstractions, not concretions — high-level and low-level modules should both depend on interfaces.	solid
What is a design pattern?	A reusable, named solution to a commonly occurring problem in software design — e.g. Singleton, Factory, Observer.	patterns
Singleton pattern in one sentence.	Ensures a class has exactly one instance and provides a global access point to it.	patterns
Factory pattern in one sentence.	Defers object creation to a method/class so callers don't depend on concrete types — pick the implementation behind a common interface.	patterns
Observer pattern in one sentence.	A subject maintains a list of observers and notifies them automatically of state changes — basis of pub/sub and event systems.	patterns
Strategy pattern in one sentence.	Encapsulates interchangeable behaviors behind a common interface so the algorithm can be swapped at runtime.	patterns
Adapter pattern in one sentence.	Translates one interface into another so otherwise incompatible classes can work together.	patterns
Coupling vs. cohesion.	Coupling = how much modules depend on each other (lower is better). Cohesion = how related the responsibilities inside a module are (higher is better).	concepts
When designing a system in an interview, what's a good first step?	Clarify requirements and scope — list use cases, expected scale, and constraints before drawing classes.	interview-tips
"is-a" vs "has-a" relationships.	"is-a" → inheritance (a Dog is an Animal). "has-a" → composition (a Car has an Engine).	concepts
Why is multiple inheritance often avoided?	It introduces ambiguity (e.g. the diamond problem) and tight coupling. Many languages (Java, C#) disallow it for classes and use interfaces instead.	concepts
What is dependency injection?	Providing a class's dependencies from outside (constructor, setter, or method argument) rather than the class creating them itself — improves testability and flexibility.	concepts
What is duck typing?	"If it walks like a duck and quacks like a duck…" — a runtime style (e.g. Python) where an object's suitability is determined by its methods/behavior, not its declared type.	concepts
Abstract method vs. concrete method.	Abstract: declared without implementation, must be overridden by subclasses. Concrete: has an implementation, can be inherited as-is or overridden.	concepts
